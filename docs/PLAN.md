## 将来やること（MVP 後のバックログ）

- [ ] 古いゲストの掃除スクリプト（30日以上ログインのない匿名ユーザーを Auth から削除し、Firestore の `users/{uid}` 以下も削除。手動実行から始め、必要なら定期実行に）
- [ ] 保存後の記録編集（6項目の編集→再ベクトル化。成形カード UI を再利用）
- [ ] 参考記録の類似度しきい値（無関係な質問のとき refs に無関係な記録が付くのを防ぐ）
- [ ] 公開範囲を広げる場合の IP レート制限・ボット対策
- [ ] **保存時暗号化（他人に使ってもらう前に必須）**：日記・相談・プロフィールのテキストを AES-256-GCM でサーバー側暗号化してから Firestore に保存（鍵は環境変数。AI 処理の瞬間だけ復号）。Console やバックアップでは暗号文しか見えなくなり、DB 漏洩にも耐える。※embedding は検索の仕組み上、平文のまま。※管理者が鍵を使えば復号は可能＝「見ない仕組み」であって完全 E2EE ではない（完全 E2EE は AI 分析と両立不可）。実装時は `enc:v1:` プレフィックスで平文とのフォールバック互換を確保し、事前にフルバックアップ＋鍵の控えを別所に保管。半日仕事
- [ ] 記録データ取得のスケール対応（発動条件の目安：Firebase 無料枠は読み取り5万回/日・容量1GiB。最初に苦しくなるのは読み取り回数で、数十人が毎日使う規模から。超えそうなら Blaze プラン切替＋以下を実装。DB 費用は誤差で、支配的コストは AI 側な点に注意）：①ユーザードキュメントに記録日一覧 `recordDays` を保存時に積み、ホームは「日付一覧＋直近5件」だけ取得（現状は全記録の全文を一括取得・上限2000件）②カレンダーの日付タップはその日の分だけオンデマンドでクエリ ③相談の RAG も全記録＋embedding を毎回読む構造のため、同時期に Firestore Vector Search へ移行を検討

## 進め方のルール

- タスクを完了したらチェックを付け、そのタスク名でコミットする（例：`feat: 深掘り API Route と質問バブルを実装`）

- 画面を仕様から変えた場合は mockups/ と docs/ を同時に更新する（CLAUDE.md のモック更新手順）
