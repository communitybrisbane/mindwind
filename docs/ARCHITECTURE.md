# MindWind - 技術設計書

## 技術スタック

- **Frontend**: Next.js（最新安定版・App Router）+ TypeScript + Tailwind CSS
- **Backend**: Next.js API Routes（Firebase Admin SDK）
- **AI**: Claude API（深掘り質問=Haiku 4.5／成形=Sonnet 5／相談=Opus 4.8。詳細は「使用モデルと API 機能」参照）
- **Embeddings**: OpenAI `text-embedding-3-small`（1536次元・ベクトル化）
- **Database**: Firebase Firestore（記録・ベクトル保存）
- **Authentication**: Firebase Authentication（Google ログイン、MVP から実装）
- **Audio**: Web Speech API（音声入力。iOS Safari の自動停止には onend 自動再開ループで対応。非対応環境はマイク非表示。詳細は DESIGN-mindwind.md）
- **Deploy**: Vercel

## 認証・セキュリティ設計

- スタート画面（`/`）の「はじめる」ボタン = Google ログイン。成功後 `/home` へ遷移
- クライアントは Firebase ID トークンを API リクエストに添付（`Authorization: Bearer <token>`）
- API Routes は Firebase Admin SDK でトークンを検証し、`uid` に紐づく記録のみ操作
- Firestore セキュリティルール: `request.auth.uid == userId` で本人以外のアクセスを遮断
- Claude / OpenAI の API キーはサーバー側のみで使用（クライアントに露出しない）

## ベクトル検索の実装方法（MVP）

### MVP での実装：シンプル実装

- 記録保存時、マージ後の6項目（出来事・思考・行動・理由・感情・価値観）を連結したテキストを OpenAI `text-embedding-3-small` でベクトル化
- Firestore に `embedding` フィールド（number 配列・1536次元）で保存
- 相談時は API Routes（Admin SDK）が本人の全記録を読み込み、コサイン類似度で上位5件を抽出
- 抽出した記録を Claude に渡してパターン分析・アドバイス生成

### 特徴

- 実装がシンプル（追加 DB 不要）
- 数百件程度のデータで十分パフォーマンス出力可能
- 新卒向けアプリの初期ユーザー数に対応可能

### 将来：Firestore Vector Search への移行

- データが増えてパフォーマンス問題が出たら、Firestore Vector Search にネイティブ移行可能

## 使用モデルと API 機能

機能ごとに要求（レイテンシ・構造化・知性）が異なるため、モデルを使い分ける。

| 機能 | モデル | 理由と設定 |
|------|--------|-----------|
| 深掘り質問の生成 | `claude-haiku-4-5` | 短い出力・単純タスク・低レイテンシ優先（日記送信直後にユーザーを待たせる場所）。thinking なし、`max_tokens` は小さく（〜512） |
| 成形（6項目への分類・統合） | `claude-sonnet-5` | **Structured Outputs**（`output_config.format` に JSON スキーマ）で6項目＋タイトルのスキーマ準拠を保証。JSONパース失敗のエラー処理が実質不要になる。`output_config.effort: "low"` |
| 相談（RAG＋パターン分析＋アドバイス） | `claude-opus-4-8` | アプリの価値の核心で最も知性が要る処理。`thinking: {type: "adaptive"}` を明示（Opus 4.8 は省略すると thinking なしになる）、`effort` はデフォルト（high） |
| ベクトル化 | OpenAI `text-embedding-3-small` | 安定・安価（変更なし） |
| 音声入力 | Web Speech API | ブラウザ完結・無料（変更なし） |

### 相談 API の必須機能

- **Streaming**: 回答をトークン単位でチャット UI に流す（`messages.stream`）
- **Prompt caching**: プレフィックスを「固定システムプロンプト（メンター人格＋プロフィール注入文）→ スレッド内履歴」の順に組み、履歴の最後に `cache_control` を置く。**可変部分（RAG記録5件＋新しい質問）は必ず末尾**に置く（プレフィックスに入れるとキャッシュが毎回無効になる）。2ターン目以降の実効入力コストは約1/10になる

SDK は TypeScript（`@anthropic-ai/sdk`）を Next.js API Routes から使用。API キーはサーバー側のみ。

## Claude へのプロンプト設計

詳細は `docs/mentor-reference.md` を参照。

### 0. 共通人格プロンプト＋ユーザープロフィールの注入（全プロンプト共通）

すべての AI 呼び出しのシステムプロンプト冒頭に、**AI メンターの共通人格**（対等な相棒・記録は一例として提示・問い詰めない・裁かない・セーフティ。詳細は `docs/mentor-reference.md`「AI メンターの人格・口調」）を注入する。続けてユーザープロフィールを注入する。**プロンプトの実文（共通部＋機能別追記）は `docs/mentor-reference.md`「メンタープロンプト（実文）」に収録**しており、実装時は一字も変えずに使用する（prompt caching の固定プレフィックス要件）。

#### ユーザープロフィールの注入

`profile` の内容を1〜2文の固定文としてシステムプロンプトに注入する（例：「ユーザーの呼び名は◯◯。機械系の大学院生で、報連相がうまくなりたいと思っている」）。

- 深掘り質問・相談の回答が、ユーザーの立場（学生／年次・仕事内容・目標）を前提としたものになる
- 未入力の項目は文に含めない。プロフィール全体が未設定でも動作は変わらない
- ユーザーごとに安定した文字列なので prompt caching のプレフィックスに含められる

### 1. 深掘り質問プロンプト（現在志向）

ユーザーが送信した日記テキストに対して、必ず1回だけ質問を生成。曖昧・不足している部分を掘り下げ、より濃密な記録が残る質問を選ぶ。
- 優先度1：不足情報を具体化
- 優先度2：現在の心理状態を掘る
- 優先度3：矛盾する感情の優先度を聞く

### 2. 成形プロンプト（分類・統合。サマリーは生成しない）

深掘り回答の後に1回の呼び出しで、**日記テキスト＋深掘り回答を統合して6項目＋タイトルに成形（分類）**する：
- 出来事、思考、行動、理由、感情、価値観（価値観のみ AI 推測。UI 上のラベルは「価値観（AIの気づき）」と表記）
- 深掘り回答の情報は該当フィールド（感情・理由など）に反映し、各項目を具体的で濃い最終値にする（例：感情「心に余裕があった」→「心に余裕があった。前日までに資料を作り終えていて時間に追われず、先週の成功で気持ちも落ち着いていた」）
- **タイトルも同時に生成**：15文字以内・体言止め（例：「プロジェクトをまとめられた」）。ホームの記録一覧・相談の「参考にした記録」リストに表示する。空だった場合のフォールバックは出来事フィールドの先頭20文字。タイトルはベクトル化に含めない（表示専用）
- 質問がスキップされた場合は日記テキストのみから成形
- 成形結果はチャット内カードとして表示され、ユーザーが編集した値が保存される最終値
- 6項目の最終値がそのまま閲覧用データ：ダッシュボードの展開表示に6項目をラベル付きで表示（見返したときに各項目だけで意味が分かる）
- サマリー（自然文への統合）は生成しない。ベクトル化は6項目を連結したテキストから行う

### 3. パターン分析プロンプト

複数の記録から判断基準を認識。
- 行動パターン → 心理パターン → 価値観パターン → 成功・失敗パターン

最終出力：「Yes/No の明確なアドバイス」

## データモデル（Firestore）

```
users/{userId}/
  ├── profile: map                  # プロフィール（オンボーディングで入力・すべて後から編集可）
  │   ├── nickname: string          # 呼び名
  │   ├── activity: string          # いましていること（仕事や学業。例：営業／大学生）
  │   ├── stage: string             # いまのステージ："student" | "year1" | "year2" | "year3plus"
  │   └── goal: string              # いまの目標・課題（任意）
  ├── thoughts/{thoughtId}/
      ├── date: timestamp
      ├── title: string             # 成形時に AI が生成（15文字以内・体言止め。表示専用でベクトル化しない）
      ├── rawText: string           # 元の入力全文
      ├── event: string             # 出来事（以下6フィールドはユーザー編集＋深掘り回答をマージした最終値）
      ├── thinking: string          # 思考
      ├── action: string            # 行動
      ├── reason: string            # 理由
      ├── emotion: string           # 感情
      ├── values: string            # 価値観（AI推測）
      ├── deepDiveQuestion: string  # 深掘り質問
      ├── deepDiveAnswer: string    # 深掘り回答（生データとして保持。表示には使わない）
      ├── tags: array
      └── embedding: number[1536]   # マージ後の6項目を連結したテキストのベクトル

users/{userId}/
  ├── recordChat: map               # 当日の記録チャット（復元用。日付が変わったら破棄。ベクトル化しない）
  │   ├── date: string              # "2026-07-14"（Asia/Tokyo）
  │   └── messages: array           # {role, content, type("text"|"card"), thoughtId?}
  ├── chats/{chatId}/               # 相談スレッド（履歴として保持。ベクトル化しない）
      ├── title: string             # 最初の質問から生成
      ├── createdAt: timestamp
      ├── updatedAt: timestamp
      └── messages/{messageId}/
          ├── role: "user" | "assistant"
          ├── content: string
          ├── refs: array           # assistant のみ：参考にした記録（thoughtId）のリスト
          └── createdAt: timestamp
```

## アプリシェル

- **モバイルネイティブ設計**: max-width `430px` で PC ブラウザでも中央にスマホアプリ風に表示。**デザイン基準は 390×844（iPhone 標準サイズ）**、検証最小幅は 375px
- **レイアウト**: body に `flex justify-center`、アプリシェルに `shadow` を付与してデバイスフレーム感を演出
- **ナビゲーション**: 下部タブバー（56px、3タブ：ホーム/記録/相談、アイコンのみ）。上部ヘッダー（48px、backdrop-blur）は**ホームとオンボーディングのみ**表示（記録・相談は縦スペース優先で非表示）
- **スタート画面 (`/`)**: ナビなし。ロゴ＋タグライン＋「Google ではじめる」ボタンのみ
- **ダッシュボード (`/home`)**: ストリーク、カレンダー、最近の記録
- **viewport**: `viewport-fit=cover` で iPhone ノッチ対応

## アカウント管理（`/onboarding` 編集モードの下部）

- **ログアウト**：Firebase Auth の signOut → `/` へ遷移。データは残るので再ログインで復元される
- **アカウント削除**：2段階で実行
  1. 確認ダイアログ「アカウントを削除しますか？ すべての記録・相談履歴が完全に削除され、元に戻せません」＋ 赤の「削除する」/ キャンセル
  2. 削除処理：Firestore の `users/{uid}` 以下（profile / thoughts / chats / recordChat）を全削除 → Firebase Auth のユーザーを削除 → `/` へ遷移
- **実装注意**：Auth のユーザー削除は直近のログインが古いと `auth/requires-recent-login` エラーになる。削除実行前に Google 再認証ポップアップ（`reauthenticateWithPopup`）を挟むフローにすること

## 記録フロー（3分以内）

記録は**1つのチャット画面**で完結する（入力バーは下部固定、音声対応）。

フェーズは **日記 → 深掘り → 成形** の3段階。

1. **日記**：音声/テキストで今日のことを自由に話す/書く（入力バーの上に6項目を意識した書き方ヒントを常時表示：何があった？／どう考えた？／どう動いた？／なぜそうした？／今どんな気持ち？）。送信するとチャットの吹き出しとして表示される
2. **深掘り**：AI が日記テキストの曖昧・不足している部分について、1回だけ質問をチャットバブルで返す（スキップも可能）→ ユーザーが同じ入力バーから回答（音声可）
3. **成形**：AI が日記テキスト＋深掘り回答を統合して6項目（出来事・思考・行動・理由・感情・価値観）に成形し、**チャット内にカードとして表示**。各フィールドは**その場で編集可能**（AIの解釈違いを直し、情報を補える）
4. **保存**：編集後の6項目を最終値として連結テキストをベクトル化 → Firestore に保存（質問をスキップした場合は日記テキストのみから成形。`deepDiveQuestion` / `deepDiveAnswer` は空）

### 保存後のライフサイクル

- **保存 → ホームへ自動遷移**：トースト「記録しました」を出しつつ `/home` へ遷移。ストリークの加算とカレンダーの点灯が「書いた報酬」として見える（習慣化ループ）
- **同日の複数記録は可・上限3件/日**：記録タブに戻ると当日のチャットが保存済み状態で残っている（成形カードは読み取り専用＋「保存済み」表示）。そのまま入力バーから新しい日記を書くと、深掘り→成形→保存の同じフローが走り**別の thought** として保存される。ストリーク・カレンダーは「1日1件以上」で達成（変更なし）
- **上限到達時**：3件記録済みの日は入力バーを無効化し「今日の記録は上限（3件）に達しました」と表示。記録を削除すれば再度書ける
- **記録の削除**：ホームの「最近の記録」の展開表示から削除できる（確認ダイアログあり）。削除時は thought（embedding 含む）を完全に削除し、カレンダー・ストリークを再計算する（その日の記録が0件になればカレンダーの点灯も消える）
- **保存後の編集は MVP に含めない**：編集の場は保存前の成形カード（確認・編集フェーズ）。事後に直したい場合は削除→書き直しで代替する。要望が出たら「6項目の編集→再ベクトル化」を将来拡張として追加（成形カードの UI を再利用できるため後付けが容易）
- **日付が変わったらリセット**：記録チャットは「今日」専用。Asia/Tokyo の0時を境に翌日開くと空状態に戻る。過去の記録はホームの「最近の記録」から見る
- **当日チャットの一時保持**：復元用に当日の記録チャットを `recordChat`（下記）に保存。翌日以降は破棄してよい。ベクトル化しない

## 検索・相談フロー（スレッド単位のチャット）

相談は**スレッド（1つの相談）単位のチャット形式**。スレッド内では追い質問で会話を続けられ、「新しい相談」ボタンで新しいスレッドを開始する。過去のスレッドは **ChatGPT のログのように履歴として保存**され、一覧から開いて見返したり会話を再開したりできる。**ただしスレッド間でコンテキストは引き継がない**（別スレッドの内容が回答に影響することはない）。

各ターンの処理：

1. **ユーザーが質問**：「新しいプロジェクト始めるべき？」（続けて「焦らないために何を意識してた？」のような追い質問もできる）
2. **記録数チェック**：記録が3件未満なら分析せず「まずは3日分記録してみましょう（現在◯件）」と案内
3. **ベクトル検索**：最新の質問（追い質問で主語が省略される場合に備え、スレッド内の会話文脈で補完したクエリ）をベクトル化し、関連する過去記録を上位5件抽出。**検索対象は日記の記録（thoughts）のみ。会話履歴はベクトル化せず RAG に含めない**
4. **パターン分析・回答生成**：抽出した記録 + **現在のスレッド内の全メッセージ**（Claude に渡すのは上限20メッセージ。超過時は古い順に対象から外す。画面には全件表示）を Claude に渡し、行動・心理・価値観パターンを認識して回答
5. **履歴保存**：メッセージを現在のスレッドに保存（画面を離れても復元される）

### スレッド管理と記録への誘導

- **「新しい相談」ボタン**で新しいスレッドを作成。以前のスレッドは削除されず履歴に残る
- **相談履歴**：スレッド一覧（タイトル＝最初の質問、最終更新日時順）から過去の相談を開ける。開いたスレッドで会話を再開することも可能（そのスレッドの文脈だけが使われる）
- **スレッドの削除**：履歴ドロワーの各行のゴミ箱アイコンから削除できる（確認ダイアログあり）。`chats/{chatId}` を messages サブコレクションごと完全削除。現在開いているスレッドを削除した場合はチャット画面が新規スレッドの空状態に戻る
- **チャットの内容はアプリの知識として蓄積されない**（学習しない・RAG に入らない）。そのため大事な気づきが失われないよう、**記録（日記）に残すことを促す**：
  - システムプロンプトで、相談の結論に気づきが含まれる場合は「この気づきは今日の記録に残しておこう」と回答の締めで自然に促すよう指示
  - UI 側でも「新しい相談」ボタン押下時に「この相談の気づきを記録に残しますか？」と確認し、記録タブへ誘導できる

## 公開範囲とレート制限

- **公開範囲**：招待制・allowlist は設けない。URL を知る人だけが使う非公開運用（マーケティングしない）。Google ログインさえすれば誰でも使える構造のため、コスト保護はレート制限で行う
- **記録**：上限3件/日（既存仕様）により、記録系の AI 呼び出し（深掘り＋成形）は自動的に最大6回/日/ユーザーにキャップされる
- **相談**：**30メッセージ/日/ユーザー**を上限とする。超過時は入力バーを無効化し「今日の相談はここまで。また明日話そう」と表示。カウントは Asia/Tokyo の日付単位で API Routes 側で判定
- **運用ルール**：Anthropic / OpenAI の管理画面で**月額 spend limit を設定**しておく（アプリ側のバグによる暴走時の最後の砦）

## ストリーク・カレンダーの判定基準

- タイムゾーンは **日本時間（Asia/Tokyo）固定**、0時が日付境界
- その日に1件でも記録すればその日は「記録あり」としてカウント
- ストリーク = 今日（または昨日）から遡って連続で記録がある日数

## MVP スコープ

### 含める

- Google 認証（Firebase Authentication）
- プロフィール登録（オンボーディング・スキップ可・後から編集可）
- 記録（音声/テキスト入力）
